What is a Block, Tile, Overview, Pyramids and COGs: An Experiment

Last updated on Mar 6, 2022 5 min read

I always get confused on when, what and how to use the Jargons I mentioned in the title of this blog. I aim to dump my thoughts once and for all and hopefully help others on the way.

So, this blog plot will mostly deal with how to serve a raster on web / Desktop application in faster way.

I decided to write this blog when I wanted to serve my raster on the web application given the data is stored on S3 bucket.

I read many blogs stating you should be using COGs (Cloud Optimised GeoTiff) and I was amazed by what it had to offer. While looking at creating COGs on my local machine and then storing on my bucket, I encountered these fancy jargons. Although I had known about them from when I started my GIS carrer, but you truely understand anything only when you apply it, don’t you?

To understand the above terms, let us take an example and experiment with it. I have a sentinel-2 L2A band 03 downloaded from https://scihub.copernicus.eu/dhus/. The data is in JP2000 format (Which I found is not a cloud optimised format).

on performing gdalinfo .\T43PHP_20210123T051121_B03_20m.jp2, the output gives a detailed description of the dataset. We focus on last few lines,

Band 1 Block=256x256 Type=UInt16, ColorInterp=Gray
  Description = Grayscale
  Overviews: 2745x2745, 1372x1372, 686x686, 343x343, 171x171
  Image Structure Metadata:
    NBITS=15

The block parameter represents the shape of tile (width X height) in the raster image. For eg: A Block=256X256 represents a tile of width of 256 by 256 pixels.

What is a block?

A block corresponds to a rectangular subpart of the raster. The first value is the width of the block and the second value its height. Typical block shapes are lines or group of lines (in which case the block width is the raster width) or tiles (typically squares), such as here. Knowing the block size is important when efficient reading of a raster is needed. In case of tiles, this means reading rasters from the left-most tile of the raster to the right-most of the upper lines and progressing that way downward to the bottom of the image. Source

So one thing is clear, Block and Tile are synonyms.

Also notice the output contains Overviews: 2745x2745, 1372x1372, 686x686, 343x343, 171x171.

What is an overview?

Overviews are reduced/downsampled version of the raster. Overviews can also be termed interchangably with pyramids in GIS. The concept is to reduce the dimension of the raster to facilitate faster rendering of Raster on our application. They can be build both externally and internally. Having an external overview would generate an .ovr file along which contains the information about the downsampled raster. While interal overviews are alters the exisiting file and are permanent. I found is it is not possible to build external overviews for .jp2 (JP2000) format.

That means the downloaded sentinel-2 image already has internal overviews and tiling.

When you want to pan, zoom around the raster, it helps to have overviews. They can be build at many levels (typically from level 1-18).

Therefore, tiling is spliting up the raster into multiple blocks, while overview is reducing the resolution (downsampling) of the raster. Both overview and tiles have to be tuned to improve the performance for serving/rendering raster on our application.

Using a large tile size might reduce the overall number of GET requests but it will also mean more data transfer per request. Also, having many pyramid levels can increase the request response but the data overhead to create them can turn out to be quite expensive.

Experiments

On running gdal_translate -of GTiff .\T43PHP_20210123T051121_B03_20m.jp2 b03.tif & gdalinfo .\b04.tif to convert .jp2 to .tif, it resulted in file size of b03.tif = 55.2 MB, and

Band 1 Block=5490x1 Type=UInt16, ColorInterp=Gray
  Description = Grayscale
  Image Structure Metadata:
    NBITS=15

You’d notice that the file size of the .tif is much higher than that of .jp2. Additionally we do not see any blocks or overviews for the .tif file as opposed to that of .jp2.

On running the command gdaladdo -r average .\b03_overview.tif to add interal overview to my raster, the output of the gdalinfo b03_overview.tif resulted in File size = 74.8 MB, and

Band 1 Block=5490x1 Type=UInt16, ColorInterp=Gray
  Description = Grayscale
  Overviews: 2745x2745, 1373x1373, 687x687, 344x344, 172x172
  Image Structure Metadata:
    NBITS=15

The point I am trying to convey here is that when you add an overview to the raster, there is an additional cost in terms of storage. In my opinion, the reason ESA decided to go with .jp2 format is because of lower file size since they do not have to serve the raster on the web but instead save money on storing the data. Checkout the analysis on various formats by Vincent here

Enters Cloud Optimsed GeoTiff (COG)

With the COG, you’ll be able to load the image faster and zoom in and out much smoother than you would if you were working with a regular GeoTIFF. With the GeoTIFF, the entire image needs to finish downloading before the tiles can be generated for viewing.

Cloud Optimsed GeoTiffs (COGs) are great! I use them in my work and they allow for easy and rendering of rasters on-the-fly. What makes it great is it supporting HTTP Range request. This allows to only fetch data where you are requesting it.

To create COGs using GDAL, it is as simple as typing

gdalwarp .\T43PHP_20210123T051121_B03_20m.jp2 b03_cog.tif

The result from the above command results in a file size of 87.5 MB and, by default, internal overviews and tiles. We can tune this parameter if needed. See https://gdal.org/drivers/raster/cog.html

Band 1 Block=512x512 Type=UInt16, ColorInterp=Gray
  Overviews: 2745x2745, 1373x1373, 687x687, 344x344

I had only heard positive things about COGs up until now. But I would not use it apart from my need to serve raster on the web. Reason being the file size. It is much costlier to store a COG than say, a JP2000. Vincent does a great job explaining various formats and inspired me to do my own little experiment.
